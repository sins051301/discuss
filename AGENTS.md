# PR 토론 아카이브 — AI 에이전트 가이드

## 개요

이 프로젝트는 GitHub PR 토론을 크롤링하고, **한글로 요약/분석**하여 Docusaurus 사이트에서 열람할 수 있게 합니다.

---

## PR 링크를 받았을 때의 절차

### 1단계: 데이터 수집 (크롤링)

```bash
python3 fetch_pr.py <PR_URL>
```

- PR URL 예시: `https://github.com/facebook/react/pull/31715`
- 출력 위치: `site/docs/{repo}-pr-{number}/raw.json`
- 수집 항목:
  - PR 메타데이터 (제목, 작성자, 상태, 라벨, 변경사항 등)
  - 일반 토론 댓글 (`issues/{number}/comments` API)
  - 코드 리뷰 댓글 (`pulls/{number}/comments` API)
  - 리뷰 (`pulls/{number}/reviews` API)
- 페이지네이션(100건씩) 자동 처리
- `gh` CLI 인증이 필요함

### 2단계: analysis.md 작성 (한글 요약/분석)

raw.json을 읽고 `analysis.md`를 생성합니다. **한글**로 작성하며, 아래 내용을 포함합니다:

#### analysis.md 형식

```markdown
---
sidebar_label: "{repo} PR #{number}"
sidebar_position: 1
---

# {PR 제목 한글 번역}

> **원문 PR**: {URL}
> **저장소**: {owner}/{repo} | **작성자**: @{user} | **상태**: {state}
> **생성일**: {date} | **머지일**: {date}
> **변경**: +{additions} / -{deletions} ({changed_files}개 파일)
> **라벨**: `{labels}`

---

## 한줄 요약

이 PR이 무엇인지 한 문장으로

## 배경

어떤 문제인지, 비전공자도 이해할 수 있는 쉬운 말로 설명

## 주요 쟁점

토론에서 핵심적으로 논의된 포인트들을 정리

## 결론

어떻게 마무리되었는지 시간순으로 정리

## 배울 점

오픈소스 협업, 기술 커뮤니케이션 관점에서의 교훈
```

#### 작성 규칙

1. 기술 용어는 한글 번역 후 괄호 안에 원문 표기 (예: "서버 컴포넌트(Server Components)")
2. 토론의 모든 핵심 포인트를 빠짐없이 다루되, 중복/사소한 댓글은 요약
3. 코드 예시가 핵심 논점이면 코드 블록 포함
4. 원문 PR 링크를 상단에 포함하여 원문 접근 가능하게 유지

### 3단계: 사이트 확인

```bash
cd site && npm start
```

- `http://localhost:3000` 에서 사이드바에 새 PR이 자동으로 나타남 (autogenerated sidebar)

---

## 디렉토리 구조

```
site/docs/{repo}-pr-{number}/
├── raw.json          # 크롤링 원본 데이터 (수정 금지)
└── analysis.md       # 요약/분석 (한글) — 사이트에 표시되는 유일한 문서
```

## 파일별 역할

| 파일 | 역할 |
|------|------|
| `fetch_pr.py` | PR 데이터 크롤링 (`gh` CLI 사용) |
| `site/docs/{repo}-pr-{number}/raw.json` | 크롤링 원본 (절대 수정하지 않음) |
| `site/docs/{repo}-pr-{number}/analysis.md` | 요약/분석 (한글) |

## 주의사항

- `index.md` 파일은 **만들지 않습니다** (원문은 GitHub에서 직접 열람)
- `translated.md`, `full.md` 파일도 **만들지 않습니다**
- raw.json은 수정하지 않습니다
- analysis.md 상단에 **원문 PR 링크를 반드시 포함**합니다
