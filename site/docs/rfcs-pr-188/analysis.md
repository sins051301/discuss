# 요약 및 분석

> **원문 PR**: https://github.com/reactjs/rfcs/pull/188

## 한줄 요약

"React 컴포넌트를 서버에서 실행하고 그 결과만 클라이언트에 보내자" — 2020년 말에 시작되어 2년간 커뮤니티와 격렬하게 토론한 끝에 머지된, React 역사상 가장 큰 아키텍처 변화 중 하나입니다.

## 무슨 문제였나?

웹 앱을 만들 때 항상 이런 고민이 있었어요:

"이 코드, 서버에서 돌려야 하나 브라우저에서 돌려야 하나?"

예를 들어볼게요. 날짜를 예쁘게 보여주려고 `date-fns`라는 라이브러리(83KB)를 씁니다. 이 코드가 사용자의 브라우저로 다운로드됩니다. 근데 사실 이 라이브러리가 하는 일은 "2024-01-15"를 "1월 15일"로 바꾸는 것뿐이에요. 그 결과 텍스트만 보내면 되는데, 83KB짜리 라이브러리 전체를 사용자 폰에 다운받게 하고 있었던 거죠.

또 하나의 문제는 **워터폴**이에요. 컴포넌트 A가 렌더링되고, A 안의 컴포넌트 B가 데이터를 요청하고, B 안의 C가 또 데이터를 요청하고... 이 모든 요청이 사용자의 브라우저에서 서버로 왕복하면서 점점 느려지는 거예요. 마치 식당에서 메뉴판을 보고 → 주문하고 → 나오면 반찬을 추가 주문하고 → 그게 나오면 디저트를 주문하는 것처럼요. 한 번에 다 주문하면 되는데 말이죠.

## 어떻게 해결했나?

React 팀이 제안한 건 **컴포넌트를 두 종류로 나누자**는 겁니다:

1. **Server Components**: 서버에서만 실행됩니다. 데이터베이스 접근, 파일 시스템 읽기, 무거운 라이브러리 사용 등을 서버에서 처리하고, 그 결과(일종의 "미리 구워진 UI 레시피")만 클라이언트에 보냅니다. 번들 사이즈에 포함되지 않아요.

2. **Client Components**: 기존 React 컴포넌트와 동일합니다. 상태(`useState`), 이벤트 핸들러, 브라우저 API 등을 사용할 수 있어요.

이 둘 사이의 통신은 **특수한 스트리밍 프로토콜**("슬롯이 있는 스트리밍 JSON")로 이루어집니다.

## 토론에서 뭐가 쟁점이었나?

이 RFC는 2020년 12월부터 2022년 10월까지 약 **2년간** 토론이 이어졌습니다. 주요 쟁점들:

### 쟁점 1: "Node.js에만 묶이는 거 아닌가?" (런타임 종속성)

@Daniel15, @markerikson 등이 제기. Python, Java, Go 등 다른 백엔드를 쓰는 팀은 어떻게 하냐는 질문.

**React 팀 입장**: 현재는 JS에 집중하고 있고, 프로토콜이 안정화되면 다른 언어 지원도 고려할 수 있다.

### 쟁점 2: "왜 async/await를 안 쓰나?" (데이터 페칭 패턴)

@dfabulich가 가장 강하게 비판. "Promise를 throw하는 패턴"이 이해하기 극도로 어렵다고 주장.

**결론**: 커뮤니티의 지속적인 피드백을 수용하여, **2022년 머지 시 Server Components에 네이티브 async/await 지원을 추가**. 커뮤니티의 승리.

### 쟁점 3: "복잡성이 너무 높아지는 거 아닌가?" (DX 우려)

@ShanonJackson이 대표적. 기존에 1종류 컴포넌트만 생각하면 됐는데, 이제 서버/클라이언트/이소모픽 3종류를 생각해야 한다.

**React 팀 입장**: 강제가 아니라 선택이다. 기존 컴포넌트는 그대로 작동한다.

### 쟁점 4: "CC에서 SC를 import하면 안 되는 이유?"

@brillout이 주도적으로 파고듦. React 팀의 설명 후 납득: "이 제약이 프레임워크 복잡성을 대폭 줄여준다는 걸 깨달았다"

### 쟁점 5: 보안 (XSS)

JSX를 네트워크로 보내면 인젝션 공격에 취약하지 않냐? **React 팀**: 스트리밍 프로토콜이 인젝션 공격을 방어한다.

## 결론은?

- **2020-12-21**: RFC 최초 제출
- **2020-12-21 ~ 2021-01**: 가장 활발한 토론 기간. 200개 가까운 댓글
- **2022-10-25**: **머지** — `.server.js`/`.client.js` → `'use client'` 지시어 변경, async/await 지원 추가

두 변경 모두 커뮤니티 피드백을 직접 반영한 결과입니다.

## 배울 점

1. **큰 변화에는 인내심 있는 설명이 필수다**: gaearon과 josephsavona는 같은 질문이 반복되어도 매번 정중하고 상세하게 답변했어요.

2. **커뮤니티 피드백이 실제로 설계를 바꾼다**: async/await 지원 추가와 파일명 규칙 변경은 모두 커뮤니티의 지속적인 피드백에서 비롯되었습니다.

3. **제약의 이유를 이해하면 설계가 보인다**: "CC에서 SC를 import하면 안 된다"는 제약은 처음에 불편해 보이지만, state 직렬화 문제를 이해하면 납득이 됩니다.

4. **한 번에 완벽할 필요 없다**: 이 RFC는 명시적으로 "열린 연구 영역"을 두고, 첫 번째 반복으로 머지했습니다.

5. **복잡성에 대한 우려는 항상 진지하게 받아들여야 한다**: 새 기능이 기존 사용자에게 부담이 되면 안 됩니다.
