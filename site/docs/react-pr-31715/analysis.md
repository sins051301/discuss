# 요약 및 분석

## 한줄 요약

React의 서버 컴포넌트(Flight) 내부에서 **"아직 에러가 아닌데 에러 처리가 호출되는 문제"**를 고치고, 중복된 에러 처리 코드를 하나로 합친 PR입니다.

## 무슨 문제였나?

비유하자면 이런 겁니다. 공장 컨베이어 벨트에서 물건이 흘러가고 있는데, 벨트를 일시 정지(halt)했더니 "물건이 망가졌다!"는 알람이 울리는 겁니다. 물건은 멀쩡한데 말이죠.

React의 Flight(서버 컴포넌트 스트리밍 시스템)에서 스트림을 중단할 때, `onError`와 `onPostpone` 콜백이 불필요하게 호출되고 있었습니다. 중단은 에러가 아닌데 에러로 취급하고 있었던 거죠. 게다가 이 과정에서 에러의 "digest"(에러 식별 정보)가 사라져버리는 부작용도 있었습니다.

또 다른 문제는 코드 구조였습니다. thenable(Promise 같은 것)과 스트림에서 에러가 발생했을 때의 처리 로직이 여기저기 흩어져 있었습니다. 같은 일을 하는 코드가 여러 곳에 중복되어 있으니 유지보수가 어렵고 버그가 생기기 쉬운 구조였습니다.

## 어떻게 해결했나?

**두 가지 작업**을 동시에 진행했습니다:

1. **중단(halt) 시 에러 콜백 호출 제거**: 스트림이 중단될 때 `onError`/`onPostpone`를 호출하지 않도록 수정했습니다. "멈추는 것"과 "에러가 나는 것"을 명확히 구분한 겁니다.

2. **에러 분기 통합**: 여러 곳에 흩어져 있던 에러 처리 로직을 `erroredTask`라는 하나의 함수 아래로 모았습니다. 코드가 깔끔해지면서 번들 사이즈도 자연스럽게 줄었습니다.

결과적으로 코드 78줄 추가, 168줄 삭제 — **순삭감 90줄**. 기능을 개선하면서 코드량은 줄인 깔끔한 리팩토링입니다.

## 토론에서 뭐가 쟁점이었나?

이 PR은 토론이 거의 없었습니다. sebmarkbage(React 코어 팀 멤버)가 올리고, gnoff(역시 코어 팀)가 빠르게 승인했습니다. 코어 팀 내부에서 이미 합의된 방향이었던 것으로 보입니다.

봇 리포트에서 주목할 점은 **번들 사이즈 감소**입니다. Flight 서버 관련 번들이 2~3.5 kB씩 줄었는데, 이는 에러 분기 통합으로 중복 코드가 제거된 자연스러운 결과입니다.

## 결론은?

- PR 올린 당일에 승인 & 머지 완료
- React DOM 등 주요 번들에는 영향 없음
- Flight(서버 컴포넌트) 번들만 가볍게 사이즈 감소
- 에러 발생 시 task를 할당하지 않는 일부 케이스는 후속 작업으로 남겨둠

## 배울 점

1. **"중단"과 "에러"를 혼동하지 말 것**: 시스템에서 graceful shutdown과 error는 명확히 다른 개념입니다. 둘을 같은 경로로 처리하면 의도치 않은 부작용(digest 유실 등)이 발생할 수 있습니다.

2. **코드 중복을 줄이면 번들도 줄어든다**: 같은 로직의 에러 분기를 통합하는 것만으로 2~3 kB의 번들 사이즈를 절약했습니다. 리팩토링이 성능 최적화가 되는 좋은 사례입니다.

3. **점진적 개선**: "에러 발생 시 task를 할당하지 않는 케이스"는 이 PR에서 다루지 않았습니다. 한 번에 모든 것을 고치려 하지 않고, 범위를 제한하여 리뷰하기 쉽게 만든 점이 좋은 PR 습관입니다.
